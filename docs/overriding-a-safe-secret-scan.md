# Overriding a safe secret-scan finding

Sometimes the secret scan flags a value that is genuinely safe to publish: a throwaway self-signed cert generated for a lab, a dummy token in an example config, a fake password in a demo script. This is how to clear the finding without turning the scan off for everyone.

## First, confirm it is actually safe

The scan is a backstop, so treat every finding as real until you have proven otherwise. Before you allowlist anything, check that the flagged value:

- grants access to nothing real: no production system, account, or paid service,
- is sample material that attendees regenerate or replace when they run the lab, and
- would cost nothing if a stranger copied it out of the repo.

A throwaway cert the lab tells people to regenerate meets all three. A "test" token that still works against a real API does not. Remove that one and rotate it.

If you cannot say yes to all three, it is not safe. Remove the value, and rewrite history if it already landed in a commit.

## Then allowlist it in `.gitleaks.toml`

Once you have confirmed a value is safe, add a scoped entry to [`.gitleaks.toml`](../.gitleaks.toml) in the same PR that introduces the file. Match a single file path, not a directory or a pattern that could swallow future secrets, and write a description that says why the value is safe so the next reader does not have to re-derive it.

The Hardening Konnect lab cert is the worked example already in the file:

```toml
[[allowlists]]
description = """
Throwaway self-signed TLS material generated for the Hardening Konnect lab.
The private key is regenerated by attendees when they run the lab; it grants
access to nothing and is published deliberately as sample material.
"""
paths = [
  '''^webinars/2026-07-hardening-konnect/00-base/cert/tls\.(key|cert)$''',
]
```

Add your own `[[allowlists]]` block alongside it rather than widening the existing paths list, so each exception stays tied to its own reason.

## Check it before you push

Run the same scan CI runs, against your branch:

```bash
gitleaks detect --source . --redact --no-banner
```

`no leaks found` and an exit code of `0` means the allowlist covers your file and the scan is otherwise still clean. If you have gitleaks installed some other way, any 8.x build works. CI pins the version; your local one only needs to agree on the finding.

## Why not just override the merge

A repo admin can merge past a failed check, and for a one-off that is tempting. Skip it. The override lives only in that one merge: the finding comes back red on the next push to `main`, the next contributor sees a failure they cannot explain, and nobody can tell an approved exception from an accident. An entry in `.gitleaks.toml` is the opposite: it is reviewed in the PR, it carries its own reason, and it keeps every later run green.
